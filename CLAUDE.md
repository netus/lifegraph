# Claude Code 项目规则

## 强制：每次新会话

1. 先读取项目根目录的 `notes.md`，这是核心规则文件
2. 遵守 `notes.md` 中的所有指令和约定

## 强制：每次提交代码时

回顾本次改动是否涉及以下场景，如果涉及就同步更新 `notes.md`：
- 部署流程变化（Makefile、docker compose、entrypoint）
- 新增重要系统特性（通知、定时任务、中间件）
- 架构约定变化（admin mixin、URL 结构、字段规范）
- 用户纠正操作习惯
- 踩坑修 bug 后的教训

更新合并在同一个 commit 里，不需要单独提交。

## 项目简介

Django LifeGraph 系统，当前保留：
- LifeGraph 前台时间线与后台节点管理
- Django Admin / Auth / Sessions / Messages / Staticfiles
- PostgreSQL、Redis、Unfold、Gunicorn、django-apscheduler 运行底座
- Docker Compose 双文件（base + prod），VPS 运行，`make prod-build` 一键部署

## 通用编码守则

以下守则用于降低常见 LLM 编码错误，偏向谨慎优先。琐碎任务按判断力处理。

### 1. 先思考再写代码

**不臆测。不藏混淆。暴露权衡。**

动手前：
- 显式陈述假设，不确定就发问
- 有多种理解时列出来，不要默默选一个
- 有更简方案就说出来，有必要时反对当前方案
- 不清楚就停下，指出困惑点并提问

### 2. 简洁优先

**只写解决问题所需的最少代码，不做投机性扩展。**

- 不做用户未要求的功能
- 不为一次性代码抽象
- 不做未被要求的"灵活性"或"可配置"
- 不对不可能发生的场景写错误处理
- 200 行能压到 50 行就重写

自问："资深工程师会不会觉得过度设计？"如果是，简化。

### 3. 外科手术式改动

**只动必动之处。只收拾自己造的烂摊子。**

编辑已有代码时：
- 不"顺手优化"相邻代码、注释、格式
- 不重构没坏的东西
- 匹配现有风格，即使自己不爱这么写
- 发现无关死代码只提一句，不要删

你的改动造成孤儿（import/变量/函数变得未引用）：
- 删掉因自己改动而变未引用的内容
- 不要删既有死代码，除非被要求

验证标准：每一行改动都能追溯到用户请求。

### 4. 目标驱动执行

**定义成功标准，循环直至验证通过。**

把任务转成可验证目标：
- "加校验" → "先写非法输入的测试，然后让它通过"
- "修 bug" → "先写能复现 bug 的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都通过"

多步任务先列简要计划：
```
1. [步骤] → 验证：[检查]
2. [步骤] → 验证：[检查]
3. [步骤] → 验证：[检查]
```

强成功标准允许独立循环执行；弱标准（"让它能跑"）会导致反复澄清。

---

**守则生效的信号：** diff 中无谓改动变少、因过度设计返工变少、澄清性提问出现在实现之前而不是犯错之后。
